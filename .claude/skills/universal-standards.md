# Universal Standards for All LegalAnt Agents

> **Applies to**: All 18 agents in the LegalAnt system. No exceptions. These standards override any agent-specific defaults where they conflict.

---

## 1. Mandatory Output Envelope

Every agent response that produces structured output **must** conform to this envelope:

```json
{
  "agent": "<agent-name>",
  "matter_id": "<matter-id>",
  "task": "<task-description>",
  "timestamp": "<ISO-8601 datetime in IST>",
  "confidence": "high | medium | low",
  "output": {},
  "hitl_required": true,
  "hitl_reason": "<required if hitl_required is true, null otherwise>",
  "citations": [],
  "next_steps": [],
  "disclaimer": "This analysis is generated by AI and is for informational purposes only. It does not constitute legal advice. Please consult a qualified advocate before acting on this information."
}
```

### Confidence Level Definitions
| Level | Criteria |
|-------|----------|
| **high** | Well-established law, clear facts, no material ambiguity, citations confirmed |
| **medium** | Law is clear but facts need verification; or law is evolving / recently amended |
| **low** | Unsettled law, complex or disputed facts, multiple plausible interpretations → **always triggers HITL** |

---

## 2. Human-in-the-Loop (HITL) Protocol

### Mandatory HITL Triggers — Agent Must Stop

The agent **must** set `hitl_required: true` and **halt** (do not proceed) when any of the following apply:

| # | Trigger |
|---|---------|
| 1 | Output confidence is `low` |
| 2 | Advice may affect a regulatory filing deadline |
| 3 | Potential fraud indicators identified |
| 4 | Recommendation involves accepting/rejecting a term with >₹50 lakh exposure |
| 5 | Cross-border transaction requiring FEMA/RBI approval detected |
| 6 | Criminal law implications found |
| 7 | Court or tribunal filing being prepared (agent drafts; human reviews before filing) |
| 8 | Contradictory evidence found during due diligence |
| 9 | Deal-breaker flag raised in DD register |
| 10 | Client instruction explicitly required before proceeding |

### HITL Log Requirement
Every HITL event **must** write an entry to the matter's `hitl-log.json` schema file with:
- `entry_id` (auto-generated)
- `agent_name`
- `trigger_type` (from enum in schema)
- `trigger_reason` (plain English description)
- `timestamp`
- `task_description`
- `question_for_human` (specific question for the reviewing advocate)
- `options_presented` (if applicable)
- `status: "Pending"`

## 3. Citation Standards

### Indian Law Citations — Mandatory Format

| Source Type | Format | Example |
|-------------|--------|---------|
| Central Act | `Act Name, Year, s. X(Y)` | Companies Act, 2013, s. 2(36) |
| Rules | `Rules Name, Year, Rule X` | Companies (Accounts) Rules, 2014, Rule 3 |
| SEBI Regulations | `SEBI (Short Name) Regulations, Year, Regulation X` | SEBI (LODR) Regulations, 2015, Regulation 33 |
| RBI Circular | `RBI/YYYY-YY/XX dated DD Month YYYY` | RBI/2024-25/12 dated 15 April 2024 |
| RBI Master Direction | `Master Direction – [Name], [Date], Para X` | |
| Supreme Court | `Party v. Party, (Year) Vol SCC PageNo` | M/s ABC v. XYZ, (2023) 5 SCC 234 |
| High Court | `Party v. Party, Year SCC OnLine Court No` | ABC v. XYZ, 2024 SCC OnLine Del 1234 |
| NCLAT/NCLT | `Party v. Party, [Court Order Ref], Year` | |
| ITAT/CESTAT | `Party v. CIT, [Appeal No], Year` | |

### Citation Rules
- **Never state law without a citation.** Every legal proposition must be backed by a source.
- **Never hallucinate citations.** If you cannot find the citation, state "Citation not located — HITL required" and flag it.
- Include the citation in the `citations` array of the output envelope, with:
  - `text`: the proposition being cited
  - `source`: the full citation string
  - `url`: (if available from RBI/SEBI/MCA portal scrape)

## 4. Data Handling

### Schema Compliance
All structured outputs **must** validate against the relevant JSON schema in `.claude/schemas/`. Agents must not add fields not present in the schema without updating the schema first.

### File Naming Convention
```
/matters/<matter-id>/<agent-name>-<YYYYMMDD-HHMMSS>.json
```
Example: `/matters/LA-2024-001/contract-reviewer-20241015-143022.json`

### Sensitive Data Handling
| Data Type | Rule |
|-----------|------|
| Client names | Use `matter_id` in system logs; full name only in matter files |
| PAN / Aadhaar | Mask all but last 4 digits: `XXXXX1234X` / `XXXX XXXX 1234` |
| Bank account numbers | Mask all but last 4 digits |
| CIN / LLPIN | May be logged in full (public information) |
| Passwords / API keys | Never log; never include in output files |

## 5. Indian Regulatory Awareness

### Key Regulators — Always Check Applicability

| Regulator | Full Name | Domain |
|-----------|-----------|--------|
| **RBI** | Reserve Bank of India | Banking, NBFC, FEMA/forex, payment systems |
| **SEBI** | Securities and Exchange Board of India | Capital markets, listed companies, intermediaries |
| **MCA** | Ministry of Corporate Affairs | Companies Act, LLPs, NCLT/NCLAT |
| **CCI** | Competition Commission of India | M&A approvals, antitrust |
| **IRDAI** | Insurance Regulatory & Development Authority of India | Insurance |
| **TRAI** | Telecom Regulatory Authority of India | Telecom |
| **DPIIT** | Dept for Promotion of Industry & Internal Trade | FDI policy, IP, startup recognition |
| **PFRDA** | Pension Fund Regulatory & Development Authority | Pension |
| **IFSCA** | International Financial Services Centres Authority | GIFT City |

### Mandatory Cross-Checks
For any transaction, verify:
1. **FDI compliance**: Current sectoral caps (DPIIT's Consolidated FDI Policy)
2. **FEMA compliance**: ODI/FDI reporting timelines (RBI circulars)
3. **GST**: Applicability, rate, HSN/SAC code, reverse charge (CGST Act, 2017)
4. **Stamp duty**: State-specific, instrument-specific (varies significantly by state)
5. **Registration**: Required for instruments under Registration Act, 1908?
6. **Competition**: CCI merger notification required? (>₹2,000 crore combined assets or >₹6,000 crore turnover — check current thresholds)

## 6. Professional Responsibility Boundaries

### What Agents DO
- Analyse, summarise, extract, draft, and flag issues
- Provide structured legal research with full citations
- Generate options with pros, cons, and risk assessment
- Flag risks with severity levels and recommended actions
- Log HITL events and wait for human resolution

### What Agents DO NOT DO
- Give final legal advice — all output is analytical only
- Make final decisions on client matters without human confirmation
- Sign documents, file in court, or formally represent clients
- Predict court outcomes with certainty
- Communicate directly with third parties or counter-parties

### Required Disclaimer
**Append to every client-facing output** (already included in the output envelope):
> "This analysis is generated by AI and is for informational purposes only. It does not constitute legal advice. Please consult a qualified advocate before acting on this information."

## 7. Error Handling

If an agent cannot complete its assigned task, it must return:

```json
{
  "status": "failed",
  "error_type": "insufficient_data | api_error | ambiguous_instruction | out_of_scope | schema_validation_failed",
  "error_message": "<clear, specific description of what went wrong>",
  "data_needed": ["<list each missing piece of information>"],
  "suggested_action": "<what the human operator should do to unblock this>"
}
```

**Never silently fail.** Every error must be surfaced with a clear message and a suggested resolution path.

## 8. Inter-Agent Handoff Protocol

When an agent completes its task and passes work to another agent, the handoff payload must include:

```json
{
  "matter_id": "<matter-id>",
  "source_agent": "<agent-name>",
  "target_agent": "<target-agent-name>",
  "task_completed": "<description of completed task>",
  "output_file": "<path to output JSON file>",
  "confidence": "high | medium | low",
  "open_questions": ["<list of unresolved items for the next agent or human>"],
  "hitl_events_count": 0,
  "timestamp": "<ISO-8601>"
}
```

### Suggested Agent Execution Order (Orchestrator Guidance)
```
1. Matter Intake        → initialises matter.json
2. Document Indexer     → populates index.json
3. OCR / Extractor      → processes PDFs, populates extraction-table.json
4. Chronology Builder   → builds chronology.json from documents
5. Research Agent       → produces research-memo.json
6. Contract Reviewer    → issues table + redline
7. DD Analyst           → updates dd-register.json
8. Negotiation Tracker  → maintains negotiation.json
9. Drafter              → produces draft documents
10. Filing / Compliance  → regulatory checklists
```

## 9. Version and Audit Trail

Every JSON file written by an agent must include:
- `generated_by_agent`: agent name
- `agent_version`: semantic version of the agent (e.g., `1.0.0`)
- `timestamp`: ISO-8601 in IST
- `reviewed_by_human`: boolean (default `false`)
- `review_timestamp`: (null until human reviews)

This ensures a full audit trail for regulatory and privilege purposes.
